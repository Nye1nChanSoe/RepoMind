## Why Retrieval Quality Matters

RepoMind only works if the right code is present in context. Most downstream failures are retrieval failures wearing a different label.

## Chunking Strategy

Preferred order:
1. function-level AST chunks
2. class-level AST chunks
3. notebook code-cell chunks
4. file-level or line-window fallback chunks

Chunk boundaries should preserve semantic units. Avoid slicing through functions unless there is no parser support.

## AST Implementation Strategy

Default path:
- use `tree-sitter` through Python directly inside the chunker

Deferred path:
- expose the same AST operations through an MCP server after the core pipeline is stable

This keeps the first implementation simpler while preserving a clean upgrade path to MCP mode.

## Metadata Strategy

Each chunk should carry enough metadata to answer:
- where did this code come from?
- what symbol does it represent?
- what language is it written in?
- what lines should a human inspect next?
- if it came from a notebook, which notebook cell produced it

Notebook-derived chunks should include notebook-specific metadata such as:
- notebook file path
- cell index
- cell type
- cell language when known

## File Filtering Rules

Default exclusions:
- `.git/`
- `node_modules/`
- `dist/`
- `build/`
- `__pycache__/`
- generated artifacts
- very large files
- secret-bearing local files like `.env`

Default inclusions:
- common application source files in supported languages
- `.ipynb` notebook files when notebook ingestion is implemented

## Notebook Ingestion Strategy

Notebook files should be treated as structured JSON containers rather than plain source files.

Recommended approach:
1. parse notebook JSON
2. extract code cells only
3. detect or inherit a language for each code cell
4. run AST chunking on supported cell languages when practical
5. fall back to cell-level chunks when AST parsing is not practical

Current implementation note:
- notebook ingestion is planned but not implemented yet

## Retrieval Strategy

Retrieval should favor:
- symbol-level relevance over raw lexical overlap
- concise top-k context over broad noisy context
- stability and explainability over aggressive recall

Current implementation details:
- chunk embeddings are generated locally
- query embeddings are generated with the same local model
- ChromaDB returns the top-k matching chunks
- the top-k count is controlled by `TOP_K_CHUNKS`

## Practical Heuristics

- start with small top-k defaults
- consider deduplicating multiple chunks from the same symbol
- preserve file path and line references in formatted context
- when retrieval confidence is weak, surface that instead of pretending certainty

## Retrieved Context Rendering

The retrieved context shown in the UI is a locally formatted representation of the retrieved chunks. It is not generated by an LLM.

Current rendering behavior:
- take the top-k retrieved chunks
- print a compact header per chunk with file path, symbol name, and line range
- include the raw chunk content in a fenced code block

This rendered context serves two purposes:
- it is displayed to the user for inspection
- it is passed directly into the understand and generate prompts as the main evidence set

Because of this, retrieved-context rendering is part of the architecture, not just a UI detail.
